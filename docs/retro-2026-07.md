## Part 1 â€” Axis-field model: the gap WIN-144 didn't close

### What WIN-144 shipped

The chart *elements* now have a unified accessor-binding model (`xBinding`/`yBinding` for cartesian, `valueBinding`/`orderBinding` for slot charts), with a tween layer in `chart-context.ts` so marks read through `ctx.xGet`/`ctx.yGet` instead of raw datum fields. See `docs/axis-binding-redesign.md`.

### What it did NOT touch: the tile spec and the DockView pickers

The tile spec (`apps/sliceboard/src/persistence.ts:89-108`) still carries the *old* vocabulary:

```
measureKey?: string         // which numeric field
xKey?: string               // scatter only â€” x field
yKey?: string               // scatter only â€” y field
sortBy?: 'index' | 'value'  // two-option sort toggle
orientation?: 'vertical' | 'horizontal'
```

And `tile-sources.ts:105-109` still consumes it the old way:

```ts
const sortBy = tile.sortBy ?? 'index'
const sorted = sortBy === 'value'
  ? [...rawNodes].sort((a, b) => (b.measures[mk] ?? 0) - (a.measures[mk] ?? 0))
  : rawNodes
```

So the chart elements have a binding model, but the *tile* still emits `measureKey` + `sortBy`, and `bindTile.ts` translates via backward-compat aliases (`measureKey` â†’ `valueBinding`, etc.). The redesign stopped one layer short of the tile spec.

### The gap Winston surfaced in review

> "Picking 'Order' field is more like picking X axis field, no? Order is also asc or desc but I do not see arrow button to toggle things. Order is like, do I sort the axis by itself, as the value, somehow, and then if I do, asc or desc?"

He's right. The current `Order`/`Value` select (`DockView.ts:637-647`) is a degenerate two-option version of "pick the band-axis field." And the asc/desc arrow is just missing â€” `tile-sources.ts:109` hardcodes `b.measures - a.measures` (desc). There is no `sortDir` anywhere in the spec.

### The model that closes the gap

**For cartesian slot charts (bar, bands, line, area):** the band/x axis *is* a field choice, and the direction is a separate toggle. Concretely:

```
xField?: string         // '_index' (store order) | '_value' (sort by value) | a measure key
sortDir?: 'asc' | 'desc'   // direction; default 'desc' when xField='_value', 'asc' otherwise
valueField?: string     // replaces measureKey â€” which measure drives the value axis
```

- `xField='_index'` *is* today's `sortBy='index'` ("Order").
- `xField='_value'` *is* today's `sortBy='value'` ("Value").
- `xField='<measureKey>'` is the generalization â€” sort by an arbitrary measure. This is new capability, not a rename.
- `sortDir` is the missing arrow.
- `valueField` is `measureKey` renamed to match the cartesian vocabulary. The chart element already calls it `yBinding`/`valueBinding`; the tile spec should match.

**For scatter:** already has `xKey` + `yKey`. Rename to `xField` + `yField` for vocabulary consistency. Add `sortDir` only if we ever want to sort the x-axis (rare for scatter â€” points don't have inherent order). Probably leave scatter as-is: `xField` + `yField`, no `sortDir`.

**For radial slot charts (pie, radar, concentric-arc):** same model â€” `valueField` + `xField` + `sortDir`, where `xField='_index'` means "store order" and `xField='_value'` means "sort by value." The "x axis" here is really the angular/radial slot assignment, but conceptually it's the same: which field determines slot order.

**For hierarchical charts (pack, treemap, icicle, sunburst, treetable, tree):** `sortBy` currently means "sort leaves by value before building the hierarchy." This is a different concept from cartesian x-axis â€” it's a layout input, not an axis. Keep `sortBy` here, or rename to `leafSort` for clarity. The asc/desc arrow still applies.

### UI changes

- **Bar/bands/line/area/pie/radar/concentric-arc:** replace the `Order`/`Value` select with:
  - An `X:` field select (options: `Index`, `Value`, â€¦measure keys). Default `Index`.
  - An asc/desc toggle button (arrow icon) next to it. Default `desc` when X=`Value`, `asc` otherwise.
- **Scatter:** already has `X:` and `Y:` selects. Add the asc/desc arrow next to `X:` if we expose sort. Rename `measureKey` â†’ `valueField` in the picker label for consistency (currently scatter has no measure picker â€” it uses X/Y â€” so this is just a label change on the Y select).
- **Hier charts:** keep the sort select, add the arrow.

### Migration

This is a tile-spec change, not a chart-element change. The chart elements already have the binding model; we just need to:

1. Add `xField`, `sortDir`, `valueField` to the `Tile` interface (keep `measureKey`, `xKey`, `yKey`, `sortBy` as deprecated aliases for migration).
2. In `tile-sources.ts`, replace the `sortBy`-based sort with `xField` + `sortDir`-based sort. Emit `xField`/`valueField`/`sortDir` to the chart element via `mountProps` (the element already accepts `xBinding`/`valueBinding`).
3. In `DockView.ts`, replace the `Order`/`Value` select with the X-field select + arrow button.
4. In `bindTile.ts`, map `xField` â†’ `xBinding`, `valueField` â†’ `valueBinding`, `sortDir` â†’ sort direction in the source sort.
5. Migrate persisted tiles in `persistence.ts` load: `sortBy='index'` â†’ `xField='_index'`, `sortBy='value'` â†’ `xField='_value', sortDir='desc'`.

**This is a follow-up ticket, not part of the current retro.** The retro just names the gap.

### What's working

1. **The axis-binding redesign landed.** Chart elements have a clean accessor model. The tween layer in `chart-context.ts` is the right abstraction â€” marks read through `ctx.xGet`/`ctx.yGet`, never raw values. This is the LayerChart pattern and it's correct.
2. **Custom elements as the interop boundary.** Charts own their DOM lifecycle. Surfaces mount them directly. No React between surface and chart. This is the target architecture from `docs/dependencies.md` and it's real now.
3. **The two-lane gate (snap vs tween) is unified.** WIN-143 established it for hier charts, WIN-144 extended it to flat charts. One pattern, not per-chart special cases.
4. **`vizform-core` is right-sized.** Pure TS, zero deps, MIT. Types + colors. Doesn't need to change.
5. **`vizform-charts` is the canon.** 18 chart types, real gesture/animation infrastructure. This is the product.

### What's not working â€” refactor candidates

#### A. `demos/` is a misnomer (Winston's call-out)

`packages/vizform-charts/src/demos/` holds the production chart components â€” `MdBarChartLC`, `MdScatterChartLC`, etc. â€” exported from `src/index.ts`, consumed by sliceboard and apitable. They are not demos. The name is a leftover from when they were ported from LayerChart demos.

**Rename: `demos/` â†’ `charts/`.** The split between `demos/` (chart components) and `lib/` (shared infrastructure) is fine; just the name is wrong. `lib/` holds `chart-context.ts`, `axis.ts`, `gestures.ts`, `tree.ts`, etc. â€” shared chart infrastructure. `charts/` would hold the per-chart custom elements.

One nuance: `lib/` currently also holds `sankey.ts` and `sankey-layout.ts` (sankey rendering) and `portfolio.ts` (fixture data). `portfolio.ts` is fixture/demo data and should move to the demo harness app, not stay in the chart package. `sankey.ts` is shared between sankey variants â€” fine in `lib/`.

**Effort:** mechanical rename + import path update. Low risk.

#### B. Gen-0 packages are dead â€” verified

Checked the actual imports, not `dependencies.md` (which is stale):

- **`vizform-react-d3`**: zero consumers. No app or package imports it. Apitable's deps are `vizform-charts` + `vizform-core` + `bireactive` â€” no `react-d3`. The `dependencies.md` graph showing `apitable â†’ react-d3` is wrong.
- **`vizform-vanilla-d3`**: one consumer (sliceboard), one import (`mountTreetable` at `tile-sources.ts:13`), used only for the retired `treetable` tile kind (`tile-sources.ts:318`). The gen-0 hviz React wrappers in sliceboard (`viz/Treemap.tsx`, `viz/Sunburst.tsx`, `viz/Icicle.tsx`) use local utilities (`viz/util.ts`), not vanilla-d3 exports. `leavesOfNodes` is defined locally in both `tile-sources.ts:64` and `BrLcCharts.tsx:101`.

**Move:** delete both packages. The single `mountTreetable` import is for a retired tile kind that has an `EMPTY` schema and renders nothing useful. Drop the import, drop the retired `treetable` branch in `tile-sources.ts`, drop the retired kinds from `TileKind`.

**Effort:** small. No utility code needs to move â€” sliceboard already has its own copies of everything it uses. Just delete the packages and the one dead import.

#### C. `vizform-react-d3` duplicates `vizform-vanilla-d3`'s `viz/` directory (moot)

`vizform-react-d3/src/viz/` contains copies of `layoutTreemap.ts`, `layoutRadial.ts`, `layoutBands.ts`, `pathPrimitives.ts`, `chrome/*`, `constants.ts`, `types.ts` from `vizform-vanilla-d3/src/viz/`. Moot â€” both packages get deleted (see B).

#### D. Sliceboard's viz layer has a React membrane that's half-migrated

`apps/sliceboard/src/viz/br/BrLcCharts.tsx` (~650 lines) is a React membrane wrapping the bireactive custom elements. `bindTile.ts` is framework-agnostic. `BrLcTile.tsx` is the React tile component. The plan doc `docs/plan-dereact-sliceboard-viz.md` already targets this.

**Status:** partially migrated. The custom elements are framework-agnostic; the React membrane is a thin wrapper. The dock/persistence layer is still React (`main.ts` creates `sb-dock-view` but the dock internals use React via `store-react.ts`). Full dereact is a larger effort and may not be worth it if the React layer is thin and stable.

**Recommendation:** defer. The membrane is thin, the charts are framework-agnostic, and the dock/persistence is the real React surface. Dereacting the dock is a bigger project than the retro should kick off.

#### E. The `@svelte-lc` vite alias is still alive

`apps/sliceboard/vite.config.ts` aliases `@svelte-lc` to `apps/svelte-layerchart-spike/src/`. Verified: `SvelteLcCharts.tsx` imports 5 Svelte components (`SunburstLC`, `IcicleLC`, `PackLC`, `TreemapLC`, `Treemap`). The alias is actively used, not dead.

**Status:** the Svelte charts are a parallel authoring path for the same custom elements. The retired `svelte-br-lc-*` tile kinds suggest the bireactive ports replaced them in the default tile config, but `SvelteLcCharts.tsx` still wires them up. Need to check whether any *active* tile kind actually routes to the Svelte charts, or if this is dead code that imports but never renders.

**Recommendation:** trace whether `SvelteLcCharts.tsx` is reachable from any non-retired tile kind. If not, remove it and the alias. If yes, decide: package the Svelte components properly, or migrate the last consumers to bireactive equivalents.

#### F. `packages/vizform-react/` â€” already deleted

`dependencies.md:56` says it's a stale empty dir. Verified: it doesn't exist. Already cleaned up. The doc is stale.

#### G. Packaging smells â€” re-verified

`dependencies.md` lists several. Verified against actual `package.json` files:

1. **`vizform-apitable` has no `react` peerDependency** â€” TRUE. React is in devDeps only (`react@^17.0.2`). No `peerDependencies` section. Fix before publish.
2. **`bireactive` is app-only** â€” FALSE. `vizform-charts` already declares `"bireactive": "^0.3.4"` in deps. Stale doc claim.
3. **d3 dual-style** â€” TRUE. sliceboard pulls full `d3@^7`; packages use granular `d3-*`. Inconsistent but not broken.
4. **`*` internal version specifier** â€” TRUE. Workspace-only, not publish-ready.

**Recommendation:** fix #1 (apitable peer dep) before publish. #3 and #4 are publish-time cleanup. #2 is already done.

#### H. `lib/portfolio.ts` is fixture data in the chart package

`packages/vizform-charts/src/lib/portfolio.ts` is demo portfolio data. It shouldn't be in the chart package â€” it's not chart infrastructure. Move to the demo harness app (`vanilla-bireactive-layercharts-spike`).

### What's working but could be tighter

1. **`chart-context.ts` is doing a lot.** It holds the tween layer, the gate, the scales, the focus/zoom. As more charts use it, it may want to split into `tween-layer.ts` + `scale-context.ts` + `zoom.ts`. Not urgent â€” wait until it's a pain point.
2. **Per-chart gate logic still exists in bar/pie/radar/concentric-arc.** The redesign doc said "unified gate in chart-context" but the slot charts still have their own gate code in their `scene()`. This is because slot charts have a different tween model (slot assignment, not cartesian x/y). A shared `slot-chart-gate.ts` helper would reduce duplication. Medium effort, not urgent.
3. **`tile-config-schemas.ts` is clean.** The per-kind picker config is the right pattern. The axis-field redesign (Part 1) would add `xField` + `sortDir` picker entries here.

### Recommended refactor order

If we pursue these, the order by risk/value:

1. **Move `lib/portfolio.ts` to the demo harness.** Zero risk. Fixture data in the chart package.
2. **Rename `demos/` â†’ `charts/` in `vizform-charts`.** Mechanical, low risk, high clarity.
3. **Delete gen-0 packages (`vanilla-d3`, `react-d3`).** Small effort â€” sliceboard's only import is `mountTreetable` for a retired tile kind. Drop the import, drop the retired `treetable` branch, delete both packages. Removes ~8,300 lines of dead code.
4. **Drop retired tile kinds from `TileKind`.** Pairs with #3. Migration in `persistence.ts` load.
5. **Fix `vizform-apitable` peer dep.** Add `peerDependencies: { react: ">=17" }`. Required for publish.
6. **Trace `@svelte-lc` alias reachability.** `SvelteLcCharts.tsx` imports 5 Svelte components â€” check if any active tile kind routes to them. If dead, remove the file + alias + spike app. If alive, decide on package vs migrate.
7. **Axis-field tile-spec redesign (Part 1).** Separate effort, not a refactor. New capability.

### What not to do now

- **Don't dereact the sliceboard dock/persistence layer.** It's a bigger project than it looks, the React membrane around charts is already thin, and the charts are framework-agnostic. Defer.
- **Don't split `chart-context.ts` preemptively.** Wait until it's a pain point.
- **Don't unify the slot-chart gate logic yet.** The per-chart gates work and are tested. A shared helper is speculative until we have a third slot chart that needs it.
- **Don't switch to pnpm/turbo.** npm workspaces work fine for this size. No pain point.

## Summary

WIN-144 shipped the chart-element side of the axis-binding model. The tile-spec side (Part 1) is the next logical unit of work â€” it closes the gap Winston identified and adds the missing asc/desc arrow. It's a focused change: tile spec + tile-sources + DockView pickers + bindTile mapping.

The repo reorg (Part 2) is lower urgency but high clarity value. The single most impactful move is deleting gen-0 (`vizform-vanilla-d3` + `vizform-react-d3`) â€” verified dead, not just "legacy." `react-d3` has zero consumers; `vanilla-d3` has one import for one retired tile kind. Removing them deletes ~8,300 lines and clarifies that `vizform-charts` is the canon. The `demos/` â†’ `charts/` rename is trivial and should just be done.
