## Where we've come from — the narrative arc

### Era 1: React → bireactive (WIN-63 → WIN-82)
Removed React from vizform. De-Reacted the sliceboard shell into a vanilla bireactive DockView (WIN-81, WIN-82, WIN-83). Charts became custom elements owning their own DOM lifecycle. The interop boundary is the custom element; no framework between surface and chart. This is the target architecture and it's real now.

### Era 2: Axis-binding redesign (WIN-144)
Unified accessor model: `xBinding`/`yBinding` for cartesian, `valueBinding`/`orderBinding` for slot charts. Tween layer in `chart-context.ts` — marks read through `ctx.xGet`/`ctx.yGet`, never raw datum fields (the LayerChart pattern). **Gap noted in retro-2026-07.md:** the tile spec still carried old vocabulary (`measureKey`, `sortBy`) — the redesign stopped one layer short of the tile. A follow-up to add `xField`/`sortDir`/`valueField` was named but never ticketed.

### Era 3: Gesture/transition state machine (WIN-315, WIN-327)
Designed the global gesture state machine + transition contract (WIN-315, done). Then shipped it (WIN-327, done — PR #168): `DataViewController` lifecycle, `Editor`/`DataView`/`Kernel.Drafts` contract, replaced `GESTURE_ACTIVE_CLASS` toggles + `gesturecommit` events with `createDataViewCell`/`dataView.settle()`, deleted legacy `gesture-state.ts`. This is the branch we're still on.

### Era 4: Hierarchical overhaul (WIN-350, done)
Ported all 5 hierarchical charts (icicle, sunburst, treemap, treetable, pack) from standalone `Diagram`-based impls (~400-500 lines each) to a shared `HierarchicalChartBase` architecture: common base class, shared behaviors (wheelEdit, keyboardEdit, tileBodyDrag, tileBodyReorder, arcBodyReorder, transitionOnUpdated, previewFullRender, mark-lifecycle), per-chart geometry modules, shared Kernel/DataView/Editor infra. 24 commits. 12 Playwright tests passing. Status captured in `wiki/2026-07-18-hierarchical-overhaul-status.md`.

Feedback rounds (1-5) in `wiki/2026-07-18-feedback-handoff.md` drove: drill transition fixes (treemap/pack/sunburst affine transform pattern matching icicle), demos reorg, label overhaul, separation/padding config, pie-as-sunburst-preset. The unifying pattern: **lay out the full tree once, compute a per-node target on drill, tween every node from current → target.**

### Era 5: Flat-chart migration (commit 621b1b42, 2026-07-19)
Migrated all flat charts off `Diagram` → `CartesianChartBase` / `RadialChartBase`. This was agent-built. Centralized CSS injection + gesture management into base classes + shared behaviors.

### Era 6: Post-migration QA sweep (2026-07-19)

### WIN-366 — the three decisions (this is the real frontier)

1. **DataViewController vs DataView+Editor+GestureCoordinator.** WIN-327 spec required "All charts use DataViewController." Shipped branch has `DataViewController` in `editor.ts:190` but **no chart instantiates it**. Hierarchical charts use a separate `DataView` class; cartesian charts use `Gesture` directly. The shipped design achieves the intent (unified gesture lifecycle, cross-tile freeze) but diverges from the letter. **Decision: (a) bless DataView+Editor+GestureCoordinator as canonical and repurpose/remove DataViewController, or (b) consolidate onto DataViewController.**

2. **bar-chart / bands CSS transitions.** WIN-327 spec required "bar-chart/bands use CSS transitions for settle/reorder." Shipped: bar-chart uses anim.clock `num()` cell tweens for ALL geometry animation (sort/reorder/measure/orientation); CSS transitions on rect attrs explicitly *disabled* (`bar-chart.ts:835-846`, `attrs: []`) to avoid double-animation with the cell tweens. `transitionOnUpdated` is kept only for gesture-active class management. bands inherits from bar (`MdBandsChartLC extends MdBarChartLC`), so it uses the same cell-tween system — **bands does NOT use d3-transition** (earlier draft of this doc was wrong; `VizRenderer.ts` no longer exists). **Decision: accept cell-tween approach (update spec) or finish CSS-transition migration.**

3. **Rule 15 amendment.** WIN-327 spec required `interaction-principles.md` Rule 15 updated. The amendment noted in `orchestration-handoff.md:34` ("value edits scale live; deferred scale is a future exception") was never applied. **Action: apply the amendment (or fold into a new rule).**

## Feedback responses

### DataViewController (line 85) — "step way back with 20-20 hindsight"
Agreed. This is the conversation to have once #168 merges. The facts: `DataViewController` (lib/editor.ts:190) wraps an Editor + the global `gestureCoordinator`. `DataView` (hierarchical/data-view.ts:159) is a separate class — Kernel subscription + Editor wrapper with draft/updateDraft/commit. Nobody calls `new DataViewController`; hierarchical charts `new DataView` and pass it to gestures as `dataView: this._dataView`. The gesture configs type it as `dataView?: DataViewController` but receive a `DataView`. So the type contract and the runtime diverged. The shipped design works (unified gesture lifecycle, cross-tile freeze via gestureCoordinator), but the naming/types are confused. Step-back question: do we want one controller class or two, and what's the right seam?

### bar-chart double-transition (line 87) — "doesn't ring a bell"
The bar-chart works well now *because* the double-animation was prevented. Two animation systems were both trying to animate rect geometry: (1) the `anim.clock`-driven `num()` cell tweens (the bireactive per-cell animation that handles sort/reorder/measure/orientation), and (2) CSS transitions on rect attributes (the `transitionOnUpdated` behavior). If both ran, you'd get fighting tweens — jitter, double-easing, wrong final values. The fix at `bar-chart.ts:835-846`: keep `transitionOnUpdated` for its gesture-active class management (suppression during wheel/keyboard edits) but disable its CSS transitions on rect attrs (`attrs: []`). The cell tweens own all geometry animation. This is a deliberate, commented departure from the WIN-327 spec's "CSS transitions for settle/reorder." The spec should be updated to explain why.

### Dead packages (line 173) — "talk me through what I'm losing"
**Nothing you'd miss.** `hotbook-react-d3` is already gone. `packages/d3` is NOT dead — `bindTile.ts` still imports from it, so deleting it would break the hotbook app's tile binding. If you want to retire `packages/d3`, the work is migrating `bindTile.ts` off it first. That's a real ticket, not a delete.

### WIN-353 treetable (line 160) — "sounds crazy, base class should support HTML"
Your instinct is right. Treetable already extends `HierarchicalChartBase` and uses the base cells. The only thing it doesn't use is `_deriveWindow`/`_deriveLayout`/`_composeStandardBehaviors` — and those are SVG-layout-specific helpers, not a fundamental incompatibility. The real question (which WIN-353 should be reframed to ask): should those helpers be refactored into a surface-agnostic core + SVG-specific layer, so treetable can share the windowing/behavior-composition logic? That's a base class design question. The current "treetable is architecturally inconsistent" framing is wrong — it's the helpers that are SVG-coupled, not treetable that's HTML-alien.
