# Reorg — 2026-07-07

A hindsight re-organization of vizform. Not a rewrite, not just cleanup —
treating the working code as the asset, but reorganizing it as if we were
doing it over: names, package boundaries, semantics, docs, dependency graph.

Supersedes `flexblox-design.md` as the active plan. The flexblox doc's
*structural ideas* (kernel as a named layer, surfaces as peers, substrate
agnosticism) inform the target shape; we realize them by lifting/moving
existing code, not greenfield.

---

## 1. The steel thread (semantic layers)

```
host (HTML / vanilla / web framework)
  → themable vanilla presentation layer
      charts · graph layout · dock — framework-agnostic by construction
        → kernel / store layer (coordination)
            defaultable — there's a default workspace, or you bring your own
              → data sources (reactive cells, datasets, notebook imports)
```

Four layers. Each layer is a seam. The kernel is the layer that's
*defaultable*: you don't have to bring one, there's a default that just
works, but you can bring your own. That's the notebook parallel — Observable
gives you a runtime by default; you don't assemble one.

## 2. The substrate spectrum (proven, not decided)

Several substrates already work in this repo. None is canon or inferior; each
is a different answer to the same question (how do rerendering, recomputation,
and relayout propagate?). The right mix is unresolved.

### Substrate shapes

| Substrate | Where | Shape | What changes | Who diffs | Conservation | Notes |
|---|---|---|---|---|---|---|
| **Fine-grained signals** | `vizform-charts` (bireactive) | `signal.get()`, `signal.set()`, `computed()`, `effect()`, `batch()` | Individual values | The substrate | Native (bireactive's multi-parent lens) | Best for multi-view sync; more moving parts |
| **Coarse-grained whole-value** | Redux-like stores | `store.get()`, `store.set(nextState)`, `subscribe(fn)` | The whole dataset | The consumer (element) does the diffing | Manual fallback | Simple mental model; D3 can diff internally |
| **D3-direct / procedural** | `vizform-vanilla-d3` (gen-0) | `chart.update(data)` — no signal | Explicit `update` call | D3 enter/update/exit | Manual | No substrate; no sync bugs; single chart owns cycle |
| **Framework-native** | Svelte experiments | Svelte runes / FLIP | Framework state | Framework | Manual fallback | Native FLIP/transitions; not portable across frameworks |
| **Observable runtime** | (research only) | Named cells, explicit `inputs`, generators | Cells fire events | Runtime | Manual | Conceptually adjacent; not a signal adapter (see §6) |

### What the kernel actually needs

The element boundary is the seam. The element says: "give me the current data
and tell me when it changes." That's it. The host provides that through
whatever substrate it chose:

- **Fine-grained signals:** `effect(() => chart.update(dataset.get()))`
- **Coarse-grained whole-value:** `store.subscribe(() => chart.update(store.get()))`
- **D3-direct:** the same as coarse-grained whole-value; the element calls `update` explicitly.

This means the kernel is not "a reactive runtime." It is "the thing that
holds the current dataset and a change-notification channel." The substrate
supplies the channel. The consumer (chart, layout, dock) does the diffing if
the substrate does not.

**Do not delete gen-0.** The gen-0 code (`vizform-vanilla-d3`) is a working
D3-direct substrate with fewer abstractions and no substrate-sync failure
mode. The question is not "is it dead?" but "which surfaces does it serve,
and can we make the element boundary work with both substrates?"

The likely answer is a **mix**: some surfaces use fine-grained reactivity
(bireactive, signals), others use coarse-grained whole-value updates (D3,
Redux-like stores). The art is the seam, not the choice.

## 3. Has someone already solved this?

**Not in this exact form.**

The academic lineage — Sketch-n-Sketch (Chugh et al., PLDI 2016) →
Bidirectional Evaluation (Mayer/Kunčak/Chugh, OOPSLA 2018) →
Patch-Reconciliation (Lubin et al., PLDI 2025) — solves **code↔output**:
you write a program that generates SVG, you drag the SVG, the program updates.
That is not our problem. Our problem is **data↔viz**: a reactive dataset shared
by multiple live charts, each editable, with coordination. The bidirectional
axis is data-to-views, not code-to-output. Multi-view coordination with
gestures, conservation, and sync is not what that literature addresses.

Industry search (2026-07-07) found DAG editors, React chart libraries, zoomable
diagram editors, and animation authoring tools. Nothing that ships bidirectional
direct-manipulation data viz with multi-view coordination.

The architecture question (coarse vs fine, procedural vs reactive) has a known
partial answer: **mix them.** Kris Zyp's granular/coarse reactivity framing,
Juris's level-mixed demos, and the PLDI 2025 patch-reconciliation paper all
point the same way: reconciliation (fine-grained patching) beats forward
re-evaluation for small targeted edits, but has overhead; forward evaluation
re-runs everything and is simpler for bulk changes. D3 is the procedural/batch
path done right. Signals are the fine-grained path. The specific combination
we are building — direct-manipulation data viz with multi-view coordination,
gesture locking, conservation, and substrate choice — does not exist as a
shipped product. The pieces exist; the assembly is ours.

## 4. Surfaces (the presentation layer)

Three real surfaces, one kernel:

1. **Charts** — direct-manipulation data viz (bars, pies, sankey, trees).
   18 chart types. Multiple substrate experiments live here (bireactive
   custom elements in `vizform-charts`, gen-0 D3 in `vizform-vanilla-d3`).
2. **Graph layout** — bireactive 2D layout (state machines, flow diagrams,
   ELKJS-style). Currently `packages/vizform-layout` at ~4k
   lines. Not a spike. User wants to keep developing this; possibly donatable
   back to the bireactive project.
3. **Dock** — layout coordination (split/group/tab/keep-alive). Currently
   in sliceboard.

The flexblox doc said "if the kernel can't serve all four [test surfaces],
it's a chart library with ideas above its station." We have two real
surfaces (charts + graph layout) and one pending (dock). That's enough to
justify the kernel as a named thing — extracted from working code, not
greenfield.

## 5. The kernel (defaultable, extracted later)

The coordination layer. Owns: cross-tile sync, gesture coordination, tween
gate, conservation, reconcile, render coordination, lifecycle. Does NOT
own: reactive primitives (substrate), rendering (surfaces), persistence
(hosts).

**Stays inside `vizform-charts` for now**, where it already works. Extracting
it to a separate package is a *later* refactor, justified when a second
surface (graph layout, or the dock) actually needs the interface. The
extraction is "lift the coordination code out of charts into a named
package," not "design a kernel and hope surfaces conform."

The flexblox doc's kernel shape (SyncHub, GestureCoordinator, TweenGate,
Conservation, Reconcile, Registry, render coordination) is the right
*target*. The path is extraction, not greenfield. But the kernel must not
assume bireactive or any particular substrate; conservation needs a manual
fallback so gen-0 / D3-direct surfaces can also use it.

## 6. Data sources (notebook-adjacent)

The layer the host provides. A "data source" isn't "load a JSON file" —
it's a reactive producer that other blocks consume. Edit the source,
downstream re-runs. That maps onto the kernel's reconcile: data is settable
at any time, patch by id.

**Observable reference (researched 2026-07-07):**
- `viewof` is the notebook model of "bidirectional block" — a cell that's
  simultaneously UI and value. Same concept as a vizform block, different
  mechanism (DOM `input` events + `.value` + generators, not signals).
- Cross-notebook imports are live and lazy — the "composable reactive data
  source" pattern exists and works. Reference for the data-source layer.
- `@observablehq/runtime` is ISC-licensed, usable standalone, cleanly
  separated from the Observable HQ product. Ideas are borrowable.
- **Not a substrate adapter candidate.** Explicit deps, async observers,
  generator-based streaming, no batch API. Adapter cost would be high and
  would fight the runtime's design. The signal consensus / D3-direct remains
  the substrate spectrum.

## 7. Target package hierarchy

Borrows the flexblox doc's structural ideas, realized by lifting/moving
existing code. **Substrate is not decided yet** — the packages below may use
bireactive, D3-direct, or both depending on what the surface needs.

```
packages/
  vizform-core      — types, colors, data-ops, tree-utils (consolidated)
                      zero deps, MIT, the shared vocabulary
  vizform-charts    — chart custom elements (18 chart types)
                      kernel coordination lives here for now (extracted later)
                      deps: vizform-core, d3-* granular, possibly bireactive
  vizform-layout    — graph layout (promoted from the "spike")
                      state machines, flow diagrams, ELKJS-style 2D layout
                      deps: vizform-core, possibly bireactive
  vizform-apitable  — APITable widget binding (AGPL leaf)
                      deps: vizform-charts, vizform-core
                      peerDeps: react >=17  ← FIX (publish blocker)

apps/
  sliceboard        — main app (dock + persistence + charts)
  docs              — astro doc site
  (spikes → examples/ or labs/ if kept, deleted if dead)
```

**What this does NOT do (deliberately):**
- No `vizform` kernel package yet. Coordination stays in `vizform-charts`
  until a second surface justifies extraction.
- No multi-backend adapters. Substrate choice is still open; bireactive is one
  proof, gen-0 is another, Svelte is a third.
- No rename of the ecosystem. `vizform` stays.

**External dependencies to ship (still TBD by substrate choice):**
- `bireactive` (patched via patch-package) — one reactive substrate option
- `d3-*` granular packages — already used
- `@apitable/widget-sdk` — only in vizform-apitable
- `react` — peerDep in apitable, devDep elsewhere
- Zero for vizform-core

## 10. Posts, not articles

The release strategy is visual posts, not launch articles. "Thing I'm
working on" energy. Show the interaction, let it speak for itself. No
voiceovers needed if the UX is right. Educate lightly — the takeaway is
"this cool thing is possible and this guy knows how to do it." Not selling.
Bits and pieces, build anticipation.

Post material (patterns that already ship, shown visually):
1. The two-lane gate — drag snaps, sort tweens. 10-second screen capture.
2. The gesture lifecycle — drag, escape, revert. One gesture at a time.
3. Custom elements as interop boundary — same chart in React, Svelte, vanilla.
4. Conservation propagation — edit a leaf, siblings redistribute.
5. Graph layout — state machine diagrams that stay live.

The kernel thesis (flexblox) is the *generalization* of these patterns
written upward — a great *second* phase after the concrete posts land.
